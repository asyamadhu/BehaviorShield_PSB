ThreatShield Architecture

ThreatShield is Layer 0 of BehaviorShield. It protects users at the entry point — before any credentials are entered — by detecting the three most common attack vectors in PSB fraud:

Phishing URLs
Typosquats, IDN homoglyphs, suspicious TLDs, PSB brand abuse, IP-based URLs. Levenshtein distance check against 20 real PSB domains.
Scam Messages
14 pattern groups covering OTP harvest, KYC urgency, lottery scams, remote access, fake refunds, SIM swap social engineering.
Fake Websites
17 DOM/TLS/network signals: form hijacking, iframe overlays, keyboard loggers, SSL anomalies, favicon mismatch, clipboard sniffing.
HOW THREATSHIELD FEEDS INTO BEHAVIOURSHIELD
When ThreatShield detects a threat, it injects suspicion points directly into the BehaviorShield session scorer (b_raw). This means:

• A user arriving via a phishing link starts the session with elevated suspicion (up to +45 pts)
• A user who scanned a CRITICAL scam SMS has their transaction layer pre-alerted (+40 pts)
• A page with form-hijacking detected immediately contributes +50 pts to the session

The combination means an attacker who faked a user onto a spoofed page AND attempts a transaction will reach Tier 3 (bank hold) much faster than either signal alone.
DESIGN PRINCIPLES
+ No external API calls — fully rule-based, works offline, zero latency added
+ Legitimate users unaffected — real PSB domains score 0, legitimate OTPs score 0
+ RBI-compliant language — "bank flags" the threat, never "you are blocked"
+ Never blocks unilaterally — warns + feeds session scorer, human-in-loop
+ Privacy-by-design — no message/URL content stored, only score returned
